# Repository Guidelines

## Project Structure & Module Organization

Shore is a Gradle multi-module Java 21 library for Reactor/Spring Boot 4.0 components. Core shared modules live in `shore-commons/` and `shore-core/`. Spring Boot starters live under `shore-starters/`, for example `shore-starters/shore-web/`, `shore-starters/shore-r2dbc/`, and `shore-starters/shore-security/`. Dependency management is centralized in `shore-bom/`. Shared Gradle build logic is in `buildSrc/`, and static-analysis rules are in `config/checkstyle/`.

Use standard Gradle source layout: production code in `src/main/java`, resources in `src/main/resources`, tests in `src/test/java`, and test resources in `src/test/resources`.

## Build, Test, and Development Commands

- `./gradlew build` runs compilation, Checkstyle, tests, and packaging for all modules.
- `./gradlew test` runs all JUnit Platform tests and generates JaCoCo reports.
- `./gradlew :shore-core:test` runs tests for one module; replace `shore-core` with another project path as needed.
- `./gradlew publishToMavenLocal` publishes artifacts locally for downstream integration checks.
- `./gradlew clean` removes generated build output.

Use JDK 21. The build fails during settings evaluation on older Java versions unless explicitly bypassed.

## Coding Style & Naming Conventions

Java code is compiled as UTF-8 with `-parameters` and lint checks for raw, deprecated, and unchecked usage. Follow Checkstyle in `config/checkstyle/checkstyle.xml`: package names are lowercase, classes/interfaces use PascalCase, methods and fields use lower camelCase, constants use `UPPER_SNAKE_CASE`, files end with a newline, and `System.out.println` is forbidden.

Prefer existing Reactor, Spring Boot autoconfiguration, and module patterns before adding new abstractions. Keep public APIs small and module boundaries clear.

## Testing Guidelines

Tests use JUnit Platform with Spring Boot Test and Reactor Test dependencies available to modules. Name tests after the unit under test, usually `ClassNameTest`, and place them in the matching package under `src/test/java`. Add focused tests for new behavior and regression tests for bug fixes. Coverage XML and HTML reports are generated by JaCoCo after test tasks.

## Commit & Pull Request Guidelines

Recent history uses Conventional Commit style, especially dependency updates such as `chore(deps): bump ...`. Keep commit subjects concise and scoped, for example `fix(core): handle missing auth token`.

Pull requests should describe the change, affected modules, test evidence such as `./gradlew build`, and any compatibility impact. Include screenshots only for documentation or UI-facing changes. Link related issues when applicable.

## Security & Configuration Notes

Do not commit credentials, signing keys, Maven Central tokens, or Codecov/Semgrep tokens. Release and publishing credentials are supplied through CI secrets and Gradle/JReleaser environment variables.
